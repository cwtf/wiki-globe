# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Wiki Globe: a real-time interactive 3D globe (CesiumJS) visualizing satellites,
flights, and shipping lanes, with an OpenStreetMap crossfade on zoom and a
click-to-Wikipedia panel. It extends beyond Earth to the Moon, Sun, and all
planets/Pluto with live ephemeris positions and Wikidata/Wikipedia article
markers. **No build step**: plain HTML/CSS/ES modules served statically.
Cesium, satellite.js, and astronomy-engine are loaded from CDN `<script>` tags
in `index.html`, not npm.

## Running it

Any static file server works (opening `index.html` via `file://` does not —
texture/module fetches need an HTTP origin):

```powershell
python -m http.server 8080
# or
npx serve -l 8080
```

Then open http://localhost:8080. The `.claude/launch.json` config
(`wiki-globe`, port 8080) runs the same thing via `preview_start`.

There is no lint step, no test suite, and no bundler/transpiler — there is
nothing to "build." Verify changes by running the app and exercising the UI
(see Verification below).

## Data pipeline commands

`data/*.latest.json` / `*.latest.geojson` files are generated by scripts in
`scripts/data/` and committed to the repo (consumed directly at runtime, no
fetch-at-build step). Regenerate/validate with:

```powershell
npm run data:update                       # regenerate all data:update:* targets
npm run data:update:country-stats         # or one target at a time
npm run data:validate                     # schema/sanity-check every generated file
```

Each `update-*.mjs` fetches from a live upstream source (World Bank, UN/LOCODE,
UCDP, Wikidata, etc.) and writes a `schemaVersion` + `meta.generatedAt`-stamped
JSON/GeoJSON file; the matching `validate-*.mjs` re-checks shape, ranges, and
row counts. Mission-supplement files (`mars-missions.json`,
`venus-missions.json`, `jupiter-missions.json`, etc.) are hand-curated, not
generated — they patch gaps in the live Wikidata results for a body.

## Architecture

### Layer model

Everything that draws on the globe is a "layer" class taking `(viewer, ...)`
in its constructor, with an `init()`/`show`/`update()`-style lifecycle wired up
in `js/app.js` (the single entry point, loaded as a module from `index.html`).
Earth-orbit layers (`js/layers/satellites.js`, `flights.js`, `shipping.js`,
`heatmap.js`, `truesize.js`) are independent and toggle via sidebar checkboxes.

### Off-Earth bodies: `BODIES` config → `BodyLayer` → per-body subclass

- `js/bodies.js` is the single source of truth for every body's metadata:
  radius, texture, dot color, Wikidata `geoGlobe` QID, ephemeris source, and
  IAU orientation coefficients (`ra`/`dec`/`w`, optionally time-varying as
  `[base, rate, "T"|"d"]` triples).
- `js/layers/body.js` (`BodyLayer`) holds the shared behavior extracted from
  the Moon/Mars: live transform composition from ephemeris + IAU rotation,
  focus/tracking camera (`lookAtTransform`, re-anchored every frame), a scaled
  "proxy" interplanetary transition for focusing distant bodies, sky dot +
  label when unfocused, lazy Wikidata/Wikipedia marker loading scoped to
  that body's globe QID, Commons-resolved mission flags, category filters,
  and CPU-projected marker positions (required at true interplanetary
  distances — see precision note below).
- `js/layers/moon.js`, `mars.js`, `sun.js` are thin subclasses/configs on top
  of `BodyLayer`. `js/layers/planets.js` (`PlanetLayer`) is the generic
  layer used for Mercury, Venus, Jupiter, Saturn, Uranus, Neptune, Pluto —
  adding a new body with no special behavior is a config-only change in
  `js/bodies.js` plus adding its key to `BODY_ORDER`/`PLANET_BODY_KEYS`.
- Only one body is ever focused at a time. `js/app.js` tracks `focusedBody`,
  suspends/restores Earth-orbit layers on focus change (without touching
  their checkboxes), and syncs the `#sel-body` dropdown and
  `document.body.dataset.focus`.
- Sidebar rows use `data-scope="earth"` / `data-scope="moon"` / space-separated
  multi-body scopes (e.g. `data-scope="sun mercury venus jupiter saturn uranus
  neptune pluto"`) in `index.html`; rows with no `data-scope` are universal.
  A row is shown only when its scope list contains the current
  `body[data-focus]` value.
- `window.__globe` (set at the end of `js/app.js` boot) exposes every
  layer/viewer instance for console debugging — add new layers to it.

Read `docs/todo/solar-system-bodies-spec.md` before touching body/planet code:
it is a living design doc recording what's shipped, the exact patterns to
reuse, known pitfalls (IAU transpose bugs, CORS-safe flag resolution, texture
longitude conventions), and what's still TODO (Galilean moons, Titan, Charon,
etc. as parent-relative children).

### Rendering at interplanetary distances (non-obvious constraints)

- `scene.camera.frustum.far` is raised to `1e13` at boot to contain Neptune;
  Cesium's logarithmic depth buffer makes this workable.
- Never tween a real camera flight across true interplanetary distances.
  Bodies beyond the Moon use a scaled "proxy" flown to a nearby fixed
  distance, then swap into the true body's tracking frame on arrival — see
  `js/layers/body.js` and §5 of the spec doc.
- Surface markers (Wikipedia article dots, mission flags) must be
  CPU-projected to world coordinates every tick with
  `Matrix4.multiplyByPoint` rather than placed via a collection
  `modelMatrix` — the latter flashes/scatters markers at true planetary
  distances due to GPU precision loss. Mars is the reference implementation.
- Wikipedia geosearch (radius-capped at 10 km) only works for Earth. Off-Earth
  bodies use a live Wikidata SPARQL query scoped by `geoGlobe` QID plus
  client-side great-circle sort for "nearest articles" — never call the
  Earth geosearch endpoint for another body.
- Wikidata's `Special:FilePath` image URLs fail CORS for WebGL billboards;
  mission flags must be resolved through the Commons API
  (`action=query&prop=imageinfo&iiurlwidth=48&origin=*`) into direct
  `upload.wikimedia.org` thumbnails first (`_resolveFlags` in `body.js`).

### Other notable modules

- `js/wiki-panel.js` — the Wikipedia article panel. `openBody(bodyName, lat,
  lon, items)` renders a pre-sorted list for off-Earth bodies (no geosearch);
  Earth uses live geosearch + Nominatim reverse geocoding for city/region/
  country context articles at larger radii.
- `js/layers/heatmap.js` — all data-overlay modes (weather grid from
  Open-Meteo, country/regional indicator choropleths, skyscraper density,
  conflict event cells), each with its own live-fetch + bundled-fallback path
  and a `LIVE`/`DATA`/`…` badge.
- `js/ais.js` / `js/ais-data.js` — live AIS shipping via aisstream.io
  WebSocket or Digitraffic REST, separate from the simulated-corridor
  fallback in `js/layers/shipping.js` / `js/demo-data.js`.
- `js/country-data.js` — bundled per-country GDP/HDI/IHDI/GNI estimates used
  as a heatmap fallback.

### Design principles carried through the whole codebase

1. **Live data first, bundled data is fallback-only** — never replace a live
   fetch with a hardcoded list; a small bundled list may back one up.
2. **Badges tell the truth**: `LIVE` (fetched), `DATA` (bundled fallback),
   idle (not loaded yet), `…` (loading). Preserve this distinction in any new
   overlay.
3. **Interaction parity across bodies**: whatever works on Earth (hover
   tooltips, click-to-articles, auto-rotate) should feel the same everywhere,
   but per-body controls (e.g. day/night toggle) are added only where they
   genuinely apply — don't cargo-cult every control onto every body.
4. Only open-source/permissively-licensed imagery is used; new textures need
   a corresponding attribution line in the sidebar `.attrib` block and README.

## Verification

There's no test suite — verify changes by running the app in a browser
(`preview_start` against the `wiki-globe` launch config) and exercising the
feature. Project-specific gotchas when doing this (from
`docs/todo/solar-system-bodies-spec.md` §10):

- `preview_screenshot` tends to time out on this app because WebGL never
  goes idle — prefer `preview_eval` against `window.__globe` for state
  inspection instead of screenshots.
- In a backgrounded preview tab, `requestAnimationFrame` is paused, so camera
  flights and per-frame `tick()`s appear to freeze — this is a throttled-tab
  artifact, not a bug. Pump frames manually (e.g. `setInterval(() =>
  viewer.render())`) if you need to force progress in an inactive tab.
- Long-lived `preview_eval` promises can time out; write results to a
  `window.__x` global and poll with follow-up evals instead of awaiting
  in-place.
- When adding a body's orientation model, verify it (sub-Earth point for the
  tidally-locked Moon, sub-solar longitude or a known-feature marker
  placement for planets) — a missing/extra matrix transpose is the most
  common bug (see `moon.js` `_updateTransform` for the reference fix).
